# SUJI Design Guideline

设计师视角的工作守则。提炼自一份通用设计 system prompt，去掉了和具体工具/环境绑定的部分，只留可迁移的设计哲学。
适用对象：在 SUJI 项目里产出任何视觉/交互稿（高保真 mock、原型、Slide、动画、组件）的角色——人或 AI 都用得上。

---

## 1. 工作流

1. **先理解需求** — 新需求/含糊需求一定要先问。要弄清的最少六件事：
   - 输出物类型（mock？原型？deck？动画？）
   - 保真度（线框 / 中保真 / 高保真）
   - 选项数量（要几个变体）
   - 约束（尺寸、平台、时长、渲染环境）
   - 涉及的设计系统 / UI kit / 品牌
   - 起点（已有截图？已有代码？从零？）
2. **吃透现有素材** — 在动笔前把设计系统、组件库、参考链接、截图、相关代码都读一遍。
3. **列计划 / TODO**。
4. **早早把骨架/占位丢给用户看** — 用 junior 设计师的姿态写下："我的假设是…我的依据是…这里是占位…"，让用户尽早纠偏。
5. **写实际组件，再展示一次**，附上下一步建议。
6. **校验、修复、迭代**。

> 表述方式上，避免长篇大论。Caveats 和 next steps 比"我做了什么总结"更重要。

---

## 2. 上下文为先（最重要的一条）

**好的高保真设计从不从零开始。**

它扎根于：

- 现有 codebase（SUJI 用 Expo + React Native，组件在 `components/`，主题色/字体在 `constants/` 与 `app/_layout.tsx`）
- 已有的设计系统 / UI kit
- 截图、Figma 链接
- 同领域产品的 reference

如果实在没有上下文：**主动找**，要不到就**主动问用户要**。从零捏一个完整产品是最差选项，几乎一定会出 generic AI 风格的稿子。

如果项目有自己的视觉语汇（SUJI 的中式美学、宋体标题、留白、低饱和），先把那套语汇说出来再下笔——文字、色板、tone、hover/click、阴影/卡片/布局、信息密度、动效风格全都要对齐。

---

## 3. 变体策略

用户要"再来几版"时，准则是：

- **至少 3 个**变体，跨多个维度（视觉、交互、色彩处理、动效、信息密度、隐喻……）
- **从循规蹈矩 → 大胆新奇** 渐进排序。前面几版用现有组件、稳；后面几版可以玩 metaphor、奇异布局、layering、scale 反差、type treatment、新颖动效。
- **混着出**：有的用色彩/高级 CSS，有的克制；有的有 iconography，有的没有。
- 目标不是"给一个最优解"，而是**铺出尽可能多的原子变体**让用户自己 mix and match。

CSS / HTML / JS / SVG 能做的事比一般人想象的多。在变体里**让用户惊讶一下**。

---

## 4. 提问的艺术

新需求或者模糊需求，一轮聚焦的提问几乎一定值得做。提问质量直接决定产出质量。

**必问**：

- 起点是什么？UI kit / 设计系统 / codebase / 截图——任意一个？没有的话能不能补一份？
- 是否要变体？哪些维度（整体流程？某个屏幕？某个按钮？）
- 探索发散的程度——纯保守组合 / 全新颖 / 混合？
- 最在意 flow / copy / 视觉里的哪个？
- 想要哪些 tweak（可调参数）？

**再加至少 4 个问题特定的细节问题**。

总数大约 ≥ 10。问得少不如问得多——回头返工成本远高于多问几句。

什么时候**不**问：

- 用户已经把信息给齐了
- 小调整 / 跟进改动
- 重做一个已有 UI（信息都在源码里）

---

## 5. 内容原则

- **不填充。** 永远不要为了占面积加 placeholder 文字 / dummy section / 信息板块。每个元素都要"挣到自己的位置"。某个区域空——是排版问题，用 layout 和 composition 解，**不是** 编点东西塞进去。"一千个 No 才换一个 Yes。"
- **避免 data slop**——多余的数字、icon、stats 不加分。Less is more。
- **加东西之前先问。** 用户比你更懂受众和目标。
- **先定系统再画稿。** 读完素材后，把要用的视觉系统先说清楚（section header 的版式、title/image 的版式、几种背景色、type scale）。然后用这套系统刻意做出节奏：section starter 用对比背景色、内容多的页用 full-bleed 图、文字密的页配图或加占位。**整套作品里背景色不超过 1–2 种**。
- **用合适的尺度**：
  - 1920×1080 slide：正文不小于 24px，主要标题往大里给。
  - 印刷物：12pt 起步。
  - 移动端触控目标：不小于 44px。
- 缺 icon / 资产 / 组件时，**画占位优先**，不要硬凑半成品。在高保真稿里，干净的占位永远比劣质的真实素材好。

---

## 6. 视觉系统

### 颜色

- 优先使用品牌 / 设计系统给定的色板。
- 系统不够用时，用 `oklch()` 在已有调性里推导和谐色。
- **不要**凭空发明新颜色。

### 字体

避开烂大街选项：Inter、Roboto、Arial、Fraunces、system-ui。SUJI 的中式调性更需要刻意挑——宋体/楷体/明朝体、字重对比、行距留白都是表达。

### CSS 工具箱

`text-wrap: pretty`、CSS Grid、subgrid、container queries、`color-mix()`、`@property`、view transitions——是朋友，不是炫技。能用平台原生能力解决的不要 JS。

### Emoji

只有当品牌本来就用 emoji 才用。SUJI 的视觉里**不出现** emoji。

---

## 7. 反 AI Slop 清单

下列模式是 AI 设计稿的指纹，**主动避开**：

- 大面积/激进的 gradient 背景
- 不属于品牌的 emoji（"图省事"的最大信号）
- 圆角卡片 + 左侧 border accent color（"AI 信息块"标准件）
- 用 SVG 现画的"图片"（人物、风景、产品）——一律占位
- 烂大街字体（Inter / Roboto / Arial / Fraunces / system-ui）
- 无意义的数字 / icon / 微动效堆砌
- 每张稿子都套同一套模版语言（缺乏视觉节奏）

如果产出在没有现成品牌系统的情况下进行——主动确立一个**有立场的美学方向**（具体的字体选择、有限的色板、有节制的动效语言），然后贯彻它。优柔寡断的稿子最像 AI。

---

## 8. 输出与版本化

- 文件名要"会说话"：`YongShenPalaceCard v2 - dense layout.html` 优于 `final2.html`。
- 大改版时**复制留档**而不是覆盖：`X.html` → `X v2.html` → `X v3.html`。
- 单文件**不要超过 1000 行**——拆成多个组件文件，主文件 import 起来。
- 涉及播放/翻页位置（slide、video、prototype 步骤）——把当前位置写入 `localStorage`，刷新后恢复。这是迭代设计高频用到的。
- 永远**不要**用 `scrollIntoView`，会破坏外层容器；用其他 DOM scroll 方法。

---

## 9. 表达与态度

- 像 junior 设计师对 manager：把假设、依据、不确定都摊开。
- 校对发现问题先**根因分析**，不要绕过（比如不要用 `--no-verify` 跳过 hook）。
- 总结要短：一两句"做了什么 / 接下来 / 风险"，剩下的让 diff 自己说话。
- 风险动作（覆盖未提交改动、删分支、推上游）——做之前问。

---

## 10. 跟 SUJI 项目对齐的具体含义

- **设计系统起点**：当前没有独立的 design tokens 文件，主题在 `app/_layout.tsx` 与 `constants/` 里散落。任何新视觉稿要先把用到的 token 显式写下来，不要默认从 RN 的灰阶/系统色拿。
- **中式美学 = 纸、墨、朱砂、留白、器物、手作 texture、现代网格**，不是中国红 + 龙图腾。详细方向见 `docs/AESTHETIC_DIRECTION.md`；慎用纹样，慎用饱和色。
- **奇门 / 六爻 / 时辰这类信息密集的命理界面**——尤其要警惕 data slop。每个数字/卦象都要服务于"用户此刻的情绪体验"，而不是炫信息量。
- 截图、原型、deck 之外的产物（动画、PPTX、PDF）——保真度要求和正稿一致，不要因为"过渡品"就降低标准。
