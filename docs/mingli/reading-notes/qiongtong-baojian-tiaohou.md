## 5. 工程落点（在不依赖原文的前提下也成立的观察）

以下几条只来自对当前代码的结构观察 + CLAUDE.md / 已有 reading-notes 已经审计的相关结论，不依赖《穷通宝鉴》原文：

1. **TIAO_HOU_YONG_SHEN 是 BaziEngine 用神选择三段优先级的最高优先项**：参见 `bazi.yongshen-priority.product-decision`（`docs/mingli/claims.json`），代码实际是"先查调候，无则按身强弱扶抑"。这意味着这张表如果有偏差，会直接污染所有命主的 yongShen 输出。
2. **数据结构 vs 古籍粒度的落差**：BaziEngine 仅记 `yong / xi / ji / desc` 四字段；《穷通宝鉴》原文实际包含"主用 / 佐 / 配 / 救 / 病药 / 取贵格"等多层语义。当前结构无法表达"丙火为主，无丙取丁；有丙无壬，富而不贵"这类条件链。Phase 2 重写时建议把 `yong` 拆成 `primary / fallback / conditional` 三段，并增加 `notesByCondition` 字段。
3. **与 `hanNuanZaoShi`（寒暖燥湿）耦合点**：调候本质就是寒暖燥湿四象的工程化映射（冬寒取火、夏燥取水、春木需水火、秋金需火炼）。如果 `RiZhuStructure / GeJuV2` 已有 `hanNuanZaoShi` 字段，应让它和 `TIAO_HOU_YONG_SHEN` 互推一致：调候表给出的用神，必须能由该月份的寒暖燥湿状态独立解释。当前没有这个互校。
4. **当前 desc 文案风险**：12×10=120 条 desc 多用"为急""最急""驱寒"之类绝对化叙事，与 SUJI 产品安全文案策略（自我观察、低风险）有张力。Phase 2 渲染层应分离"算法用神"与"用户可读句"。
