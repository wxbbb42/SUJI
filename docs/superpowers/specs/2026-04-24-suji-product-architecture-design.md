# SUJI 产品架构 · 3 个候选方案

**日期**：2026-04-24
**范围**：产品定位 + 信息架构（IA）+ 核心体验
**目标**：输出 3 个哲学上不同的候选方案，带去与合伙人讨论，从中选定方向
**作者**：通过 superpowers:brainstorming 与 Claude 协作产出

---

## 1. 背景与动机

SUJI 目前技术栈已落地（Expo RN · Supabase · 本地八字引擎 · Zustand）、原 PRD 的 4-tab 结构（今日 / 问道 / 命盘 / 静心）也已有骨架。

但随着对产品的思考深入，**"产品定位 + IA 是否最优"** 需要重新审视。这不是推翻重来，而是在真正动手把剩下的模块（情绪日记、流年、关系洞察、订阅、Widget）做上去之前，先确认我们在哪条路上。

### 1.1 这次 brainstorm 的输出不是收敛

本文档**不推荐单一方案**，而是并列 3 个可以带去讨论的候选。最终选择由创始团队共同决定。

---

## 2. 共同基底（所有方案共享）

三个方案都在下面这些前提之上构建：

### 2.1 产品定位坐标

**C 主 A 辅** —— 情绪陪伴为主 + 日常轻量留存为辅。

- **C（主）**：扩展 AI 陪伴场景，让 SUJI 成为"焦虑/迷茫时会想起的那个 app"
- **A（辅）**：每天有轻量理由打开（撕历、节气、情绪点位、Widget），让 SUJI 至少进入 daily 习惯面

**明确不是主轴**（仍可能有，但不作为驱动力）：

- B（自我认知深度 tool-driven）—— 避免把产品做成"品质版的测测"
- D（传播/社交裂变主导）—— 避免把产品做成"运势分享卡生成器"

### 2.2 两个必选根基

- **生辰八字**：所有方案里都是用户画像的基础，不管 UI 上显隐
- **AI 解读**：所有方案里都是 SUJI 区别于单纯工具型 app 的核心

### 2.3 深度内容必做

用户需要感知到"这一切背后有专业推算"——纯情感陪伴没有信任锚。三个方案的区别**不是要不要做深度内容，而是放哪里 + 以什么形态存在**。

### 2.4 AI 人格范围

三个方案的 AI 人格落在 **Y（温和伴侣）** 到 **Z（化身型）** 之间。

- **Y 温和伴侣**：固定声音，记得情绪轨迹，适度主动推送，无化身、无恋爱感
- **Z 化身型**：有名字、有简洁视觉形象、有长期"关系"感，类似 Replika 但不走恋爱路线
- **X 工具型排除**：不做"换了皮的 ChatGPT"

### 2.5 合规与语气

- 不出现"算命""预测""大师""犯太岁""破财"等词
- AI 对话附免责提示："仅供参考，如需专业帮助请咨询心理咨询师"
- App Store 分类：Lifestyle / Health & Fitness，而非 Entertainment

### 2.6 技术栈与平台

- iOS 首发（继续 Expo + RN 现状）
- Supabase 继续承担 Auth + 数据同步
- 八字算法继续本地运行（已有 80KB+ 引擎成熟）
- AI 走后端代理（不再在客户端暴露密钥——这是三个方案都必须做的技术债修复）

## 3. 方案 A · 日课型「岁吉日课」

> *"中式小仪式 × 懂你的 AI 朋友"*
> 最保守、最贴近原 PRD，落地摩擦最小。

### 3.1 核心命题

SUJI 是一个**有中式仪式感的日常陪伴 app**——早晚几分钟的小仪式，加上一个记得你的 AI 朋友。

### 3.2 首屏与 IA

- **首屏**：今日（手撕黄历）
- **Tab Bar**（4 tabs）：`今日 · 问道 · 静心 · 我的`

### 3.3 核心 Loop

每日 1–3 次，每次约 5 分钟：

```
撕历看今日干支 + AI 箴言
  → 想聊就进"问道"（可选）
  → 情绪糟可做"静心"里的呼吸/音景（可选）
  → 晚间 AI 主动道晚 push（可选）
```

### 3.4 AI 人格（Y 温和伴侣）

- 固定声音"岁吉"，**无化身、无视觉形象**
- 记得情绪轨迹、记得你说过的在意的事
- **每日早晚各 1 条**主动问候（可关闭）
- 语气可选：温暖 / 直言 / 诗意

### 3.5 深度内容放置（IA-B）

- **"我的"里**置顶一张 **《我的完整解读》** 卡片（章节化档案）
  - 第一章：你是谁（性格画像）
  - 第二章：五行气质
  - 第三章：流年节奏
  - 第四章：关系模式
- Onboarding 后自动引导通读一次；之后用户可随时翻回来
- **问道对话中**命理词汇可下划线展开（IA-C 的轻量补充）

### 3.6 静心 Tab 独立

- 24 节气音景
- 五行呼吸法
- 这是 C 主定位下"工具侧"的承载

### 3.7 商业模型

**订阅制**（¥18/月 或 ¥128/年）：

- 免费：撕历 + 基础运势 + 3 次 AI 对话/天 + 1 个主题
- Pro：无限对话 + 完整解读章节 + 流年 + 全部音景 + 全部主题

### 3.8 差异化护城河

中式美学 + 每日仪式感的**视觉体验**是主要差异化来源——3D 手撕效果、纸张纹理、节气动态配色这些细节要做到位。AI 是差异化的补充，不是主力。

### 3.9 对标心智

> 小宇宙 × Calm × 有命盘 context 的 AI 朋友

### 3.10 最适合的场景

- 团队希望**首发最稳、摩擦最小**
- 现有 PRD 基底和代码可直接复用
- 相信"日常习惯化"是留存的根基

### 3.11 主要风险

- **AI 不是主角**，如果仪式感/音景做得不够惊艳，容易被看成"又一个冥想 app"
- 差异化护城河依赖视觉品质——对前端打磨要求极高
- 合伙人可能觉得"不够有种"、没有产品赌性

### 3.12 MVP 发布范围（最少可跑）

- 今日：撕历 + AI 箴言 + 分享卡
- 问道：基础对话 + 流式输出 + 对话历史
- 我的：《我的完整解读》的"第一章：你是谁"（性格画像）
- 静心：5 个节气音景 + 1 个呼吸引导
- Onboarding + Apple/Google 登录 + 付费订阅门控

## 4. 方案 B · 知己型「岁吉小真」

> *"懂你八字的 AI 知己 × 可深挖的本命档案"*
> 最激进、最有辨识度、也最容易被划错赛道。

### 4.1 核心命题

SUJI 是一个**有具象身份的 AI 知己**——它懂你的八字、记得你的故事、在你需要时在那里。同时你可以随时打开"本命"档案看自己。

### 4.2 首屏与 IA

- **首屏**：问道（直接进对话）
- **Tab Bar**（4 tabs）：`问道 · 本命 · 静心 · 我的`
- **"今日"被降级**：作为问道顶部可折叠的小卡片，不再独占 tab

### 4.3 核心 Loop

情绪触发为主，每次 5–20 分钟：

```
打开 → AI 问候 / 你主动说话
  → 深入对话（AI 引用你的命盘作为 context）
  → 点"本命"深挖自己（可选）
  → 临睡可选做静心工具
```

### 4.4 AI 人格（Z 化身型）

- **有名字**（默认"小真"/"岁吉"，用户可自定义）
- **有视觉形象**（极简中式风格，几种可选：水墨小人 / 一盏灯 / 一片云 / 一颗星）
- 可选声音风格：温暖共情 / 直言不讳 / 诗意哲思
- 有"关系"感但**不走恋爱化**——像一个住在你手机里的朋友，不是恋人
- 长期记忆（使用 Supabase 记忆表）

### 4.5 深度内容放置（IA-A · 独立「本命」Tab）

**「本命」Tab 是 SUJI 的知识库**，也是 onboarding 后的第一个 wow 瞬间：

- 性格画像（AI 生成长文）
- 五行气质 + 十神能量分布
- 大运 / 流年（可下拉切换年份）
- 关系合盘（输入对方生辰，AI 解读相处）
- 重要时间节点提醒

### 4.6 静心 Tab

保留但**视觉上次要**——节气音景 + 呼吸工具，但不再是方案 A 里那种"主打栏"。

### 4.7 商业模型

**订阅 + 化身商店**：

- 免费：基础化身 + 每日 10 次对话 + 本命第一章
- Pro：化身进阶皮肤 + 无限对话 + 完整本命 + 流年/合盘 + 全部音景
- 未来可选：联名化身（如艺术家合作皮肤）作为附加收入

### 4.8 差异化护城河

**化身的关系感 + 命盘深度专业内容**的组合。单独做化身会被 Replika 碾压，单独做命盘会被测测碾压——两者叠加才是 SUJI 的独占地带。

### 4.9 对标心智

> Replika 的陪伴感 × 测测 Pro 的深度 × 中式哲学

### 4.10 最适合的场景

- 团队相信**"AI 陪伴是下一个留存引擎"**
- 愿意投入资源做化身系统 + 深度命盘
- 敢于在发布时承担"被贴标签"的风险

### 4.11 主要风险

- **最大风险**：化身型容易被 App Store / 用户划到**虚拟恋人/二次元赛道**，与"self-care 工具"定位冲突。需要在视觉/语气/宣发上极力避免这种滑坡
- "本命"Tab 做得不克制会滑向**传统算命 APP 味道**，违反合规红线
- 化身 + 深度命盘 = 研发量是三个方案里**最大**的

### 4.12 MVP 发布范围

- 问道：化身系统（2–3 个基础形象）+ 命名 + 长期记忆 + 流式对话
- 本命：性格画像 + 五行分布 + 完整 AI 解读报告
- 静心：3 个节气音景 + 1 个呼吸引导
- 今日卡片（折叠在问道顶部）
- Onboarding + 登录 + 付费订阅门控

## 5. 方案 C · 情绪调香型「岁吉如是」

> *"焦虑时打开的数字香薰 × 个性化情绪仪式"*
> 最贴 C 主定位，但留存数据最挑战。

### 5.1 核心命题

SUJI 不是"聊"、不是"撕"——它是**打开即走一段情绪 journey**。用户带着一种心情进来，5–10 分钟后带着被安顿好的感觉离开。

### 5.2 首屏与 IA

- **首屏**：情绪选择器（"你此刻感觉如何？"）—— 6–8 种情绪 tile（焦虑/失眠/迷茫/倦怠/低落/好奇/...）
- **Tab Bar**（4 tabs）：`此刻 · 问道 · 本命 · 我的`

### 5.3 核心 Loop

情绪触发，每次 5–15 分钟：

```
选情绪 → 系统合成个性化 journey：
  · AI 箴言（基于情绪 + 今日干支 + 你的命盘）
  · 呼吸引导或音景（时长自适应）
  · 可选 AI 倾诉（1–3 轮对话）
  · 收尾卡（生成"今日印记"，可分享）
```

**关键特征**：journey 是**系统动态合成**的，不是预制内容库。每次打开都是新组合。

### 5.4 AI 人格（Y 引路人）

- AI 以"引路人"身份**穿插在 journey 叙事中**，不是聊天主场
- 用户可以在 journey 结束后"延长对话"
- 固定声音、无化身
- 主动性极弱（不发 push，靠情绪本身驱动）

### 5.5 深度内容放置（IA-A 轻版 + IA-C）

- 独立「本命」Tab，但**内容比方案 B 更轻量**：
  - 自我画像（精简版性格 + 情绪模式图谱）
  - 不做流年、合盘、十神详解
- 对话中专业词汇可点开展开（IA-C）
- 核心哲学：**深度内容服务于情绪解读，不独立成赛道**

### 5.6 "静心"融入 journey

- **不再有独立的"静心" tab**
- 呼吸 / 音景 / 冥想都是 journey 的**组件块**
- 用户无法"单独打开一个音景听"——必须走一段 journey

### 5.7 商业模型

**订阅 + journey 解锁**：

- 免费：每日 2 次 journey + 基础本命画像
- Pro：无限 journey + 深度 journey 类型（如"梦境复盘""重要决定前""节气转折"）+ 完整自我画像 + 历史印记回顾

### 5.8 差异化护城河

**Journey 生成算法 × 命盘个性化** —— 市面上没有任何一个 app 做基于八字的情绪 journey 动态合成。这是 SUJI 独占的产品形态。

### 5.9 对标心智

> Calm 的 daily check-in × 一禅小和尚的内容叙事 × 命盘定制的 AI 引导

### 5.10 最适合的场景

- 团队相信**"情绪刚需 > 习惯养成"**
- 愿意投入内容生产和 journey 算法研发
- 不在意 DAU 数字好看，更在意"用户离不开"

### 5.11 主要风险

- **最大风险**：情绪驱动 = **低频**（心情好就不打开），留存数据会比方案 A/B 难看
- Journey 内容生产量大，**品质不稳会穿帮**——AI 生成的箴言、叙事引导需要大量调试
- **没有"日常打开"的锚**，用户可能忘记 SUJI 的存在；需要靠"节气推送"等机制补充
- App Store 截图/宣发时"卖点"最难讲清楚（方案 A 可以放撕历动画，方案 B 可以放化身，方案 C 是一个流程）

### 5.12 MVP 发布范围

- 此刻：4 种情绪入口 + journey 合成引擎（箴言 + 呼吸 + 收尾）
- 问道：AI 延伸对话（journey 后接入）
- 本命：轻量自我画像
- 今日印记：历史回顾 + 分享卡
- Onboarding + 登录 + 付费

## 7. 与合伙人讨论时可用的"收敛问题"

以下问题可用来把 3 选 1 变成有迹可循的决策过程。不需要顺序全问，挑最相关的几个聊：

### Q1 · 用户打开 SUJI 的**主要触发**是什么？
- "时间到了、习惯了" → A
- "想跟谁说说话" → B
- "现在心情很糟" → C

### Q2 · 我们的**差异化故事**想讲哪个？
- "中式仪式感 × AI 温度" → A
- "懂你八字的知己" → B
- "情绪 journey 引擎" → C

### Q3 · 研发资源（第一年）是否撑得起化身 + 深度命盘的组合？
- 撑得起 → B 可考虑
- 撑不起 → A 或 C

### Q4 · 我们在 **DAU 好看** 和 **用户离不开** 之间更看重哪个？
- DAU → A
- 长期粘性 → B 或 C

### Q5 · 三个方案里，**哪个最让你兴奋**？哪个**最让你害怕**？
- 如果兴奋和害怕同时指向同一个 → 可能就是对的那个，但需要直面风险

### Q6 · 我们能否接受"首发时被归错赛道"的短期风险？
- 不能 → 排除 B（化身最容易被误读）
- 能 → B 在桌面上

- `lib/supabase/*` + `lib/store/authStore.ts`（Auth + profile）—— 全保留

各方案**会重写/新增**的部分：

| 模块 | A | B | C |

## 9. Open Questions（留给下一轮讨论）

这些问题没在本文档里回答，需要与合伙人再讨论：

1. **用户画像最终收紧**：现在 PRD 写的是 "18-35 岁都市女性"，但在三个方案下，画像的颗粒度可以更细（焦虑型女生？文化白领？Z 世代？）
2. **AI 主动性的频率边界**：每天几条推送是"温暖"、多少条是"骚扰"
3. **商业模型的免费/付费门控具体切法**
4. **安卓 + Web 端的时间线**
5. **Widget / Siri Intent / 通知中心的优先级**
6. **合规层面**：如果选方案 B，App Store 提审时如何描述化身，避免被划到"AI Companion（恋人）"分类
7. **是否有 V0 之前的用户测试计划**（跟 10 个目标用户聊一次，快速验证 A/B/C 哪条路直觉更对）

## 10. 下一步

本文档的目的是让创始团队能**平等地看到三个方案**并做决策。

决策产出应该是：

1. **选定一个主方案**（A / B / C 或明确的混合方案）
2. **明确要保留/舍弃的元素**（比如 "选 A 但加方案 B 的化身元素"）
3. **基于选定方案写一份实施 spec**（然后进入 writing-plans 流程）

在选定之前，**不要开始大规模重构**——当前代码库可以继续做小修小补（Bug fix / 美学打磨），但不要着手 tab 结构调整或 AI 架构重写。
